# 终极研究框架：基于查询漂移的 RAG 知识库动态演进与图谱增强

## 1. 研究动机与核心痛点 (Motivation)

当前的 RAG 研究面临“知识更新”的瓶颈。现有文献大多聚焦于**“供给侧 (Supply-Side)”**，即盲目地将新文档切块入库或进行局部事实编辑。
然而，真实生产环境受限于**容量预算 (Budget Constraints)**，且面临剧烈的**查询漂移 (Query Drift)**（用户从关注 A 领域突然转向 B 领域）。
单纯的供给侧更新不仅导致信噪比极速下降，且完全无法应对领域级别的需求转移。我们需要一种**“需求侧/查询驱动 (Query-Driven)”**的知识库自适应新陈代谢机制。

---

## 2. 实验场景设定 (Experimental Setup)

构建一个连续时间流的模拟环境：

*   **文档池 (Global Pool):** 模拟海量外部世界数据。
*   **知识库 (KB):** 固定容量（如预算仅为 Pool 的 5%）。
*   **时间窗口 (Window $T$):** 用户的 Query 以数据流的形式持续输入。
*   **漂移注入 (Drift H1 $\rightarrow$ H2):** 前半段 Query 集中在旧领域，某时间点后瞬间切换至新领域（如纯单跳事实 $\rightarrow$ 复杂多跳推理，或领域A $\rightarrow$ 领域B）。

---

## 3. 全面基线对标 (Comprehensive Baselines)

为了防守审稿人可能的各种质疑，我们设计了涵盖**静态、供给侧、被动搜索、人工反馈**的 6 大 Baseline：

*   **B0: 静态底线 (Static)**
    *   *机制：* 初始化后彻底冻结。展示纯粹 Query Drift 带来的灾难性性能崩塌。
*   **B1: 盲目供给侧 (Random FIFO)**
    *   *机制：* 模拟无脑定时任务，新文档随机挤掉老文档。库内充满噪声。
*   **B2: 相似度到达 (DocArrival / 结构化RAG基线)**
    *   *机制：* (对标 HippoRAG2/LightRAG)。仅当新文档与 KB 现有内容相似时才吸纳。
    *   *痛点：* “信息茧房”效应。漂移发生时，会拒收新领域的文档。
*   **B3: 事实知识编辑 (KnowledgeEdit / 知识修正基线)**
    *   *机制：* (对标 RECIPE)。在同领域内寻找高相似度 (0.4<sim<0.8) 新文档进行 1对1 事实替换。
    *   *痛点：* 只能在旧领域修修补补，面对 H2 领域的全新需求直接瘫痪。
*   **B4: 被动现搜现用 (OnDemandFetch / Agent CRAG 基线) —— 【重要防守盾牌】**
    *   *机制：* 维持静态 KB，当遇到 Fail Query 时，立刻去外部 Pool 搜 Top-K 回答，**用完即弃，不存入 KB**。
    *   *痛点：* 在面对持续的 Query Drift 时，会导致极高的检索延迟与 Cost（API调用成本骤增）。它证明了**“将外部知识沉淀到全局 KB (Consolidation)”**的必要性。
*   **B5: 人工日志反馈 (LogDrivenArrival / Human-in-the-loop 基线)**
    *   *机制：* 模拟真实运营。在 Window $T$ 结束后，分析 Fail Query 日志，并在 Window $T+1$ 手动加入缺失文档。
    *   *痛点：* 永远存在**“滞后效应 (Lagging)”**，无法在漂移发生的当期立刻自适应。
*   **B6: 上帝视角 (Oracle) —— 理论上限**
    *   *机制：* 直接利用未来知识，将 H2 阶段的 Gold Supporting-Fact (SF) 完美注入 KB。

---

## 4. 基础方案 V1：查询驱动的动态换血 (Simple Query-Driven)

*   **机制：** 在当期窗口内，监控 Query 对 KB 的 Alignment Score，发现失效后去 Pool 拉取补给，并基于均值效用（Utility）淘汰（Evict）当前占着坑但不回答问题的“垃圾”旧文档。
*   **初步成果：** 这一步已经超越了 B1~B5 的传统范式，证明了需求侧演进的可行性。
*   **痛点暴露：** 尽管打败了现有范式，但召回率与 **Oracle (B6)** 依然存在巨大鸿沟。

---

## 5. 数据诊断：为什么 V1 离 Oracle 还有差距？(Diagnostic Analysis)

*(这是全篇最硬核、最彰显科学严谨性的部分，用数据说话)*

针对 V1 算法跑出详细诊断，核心瓶颈定位如下：

| 诊断指标                              | 数值      | 核心含义与结论                               |
| :------------------------------------ | :-------- | :------------------------------------------- |
| Query $\rightarrow$ SF 直接相似度均值 | 0.523     | 密集向量（Dense）检索对 SF 的直接捕获能力弱  |
| SF 相似度 > 0.55 (命中阈值)           | 46.9%     | 即使 SF 在 KB 里，依然有超一半检索不到       |
| Top-50 检索覆盖 SF (在全局 Pool 中)   | **86.7%** | 证明：**算法能从海量外部池中选对文档**       |
| 两个 SF 均 sim > 0.5                  | **32.8%** | **核心断点：** 仅 1/3 的查询能同时找到两篇SF |
| 仅 1 个 SF sim > 0.5                  | 51.8%     | **多跳鸿沟：** 绝大多数只够得到“第一跳”      |
| 两个 SF 均 sim < 0.5                  | 15.4%     | 完全失效的情况较少                           |

**结论总结 (The Multi-Hop Semantic Gap):**
系统并未选错文档（Top-50 覆盖率高达 87%），真正的瓶颈在于 **KB 内的匹配机制**。对于多跳查询（如“某电影的导演是谁？”），标准的纯 Embedding 余弦相似度根本跨越不了 Query 到“第二跳事实（关于导演的履历段落）”的巨大语义距离。

---

## 6. 进阶方案 V2：图谱增强与预测感知 (The Ultimate Algorithm)

为了彻底弥补上述 47% -> 67% 的 Gap，我们提出终极解法：**A + C 组合拳（图谱辅助 + 预测预取）**。

### 核心引擎 A：图谱辅助文档连带检索 (Graph-Assisted Contagious Retrieval)

*   *解决“第二跳找不到 (32.8% 痛点)”问题，巧妙呼应 HippoRAG。*
*   **机制：** 在 KB 内部维护一张**轻量级文档图谱 (Document Co-occurrence Graph)**，边由共享实体（Shared Entities）构成。
*   **动作：** 当检索系统通过 Dense Vector 勉强捞起“第一跳”文档 (Doc A) 时，不再死磕 Query 与 Doc B 的低相似度，而是直接**沿图谱边 (Doc A $\rightarrow$ 共享实体 $\rightarrow$ Doc B)** 触发连带激活，将“第二跳”事实一并拉出。

### 辅助引擎 C：漂移预测与主动预取 (Drift Prediction & Pre-fetching)

*   *解决“永远处于被动响应”的滞后问题。*
*   **机制：** 计算 Query Embedding 在时间序列上的滑动均值和变化梯度 ($\Delta$ Drift)。
*   **动作：** 在大规模 Drift 爆发的前夕（感知到分布中心开始偏移），**主动预取 (Pre-fetch)** 关联领域的文档入库，将 V1 的“滞后被动换血”升级为“前瞻性自适应”。

## 7. 预期核心贡献与故事线 (Expected Contribution & Narrative)

这篇工作的立意将非常深远，核心卖点可归纳为三步曲：

1.  **打碎旧认知：** 通过全面的实验（对比 B1~B5）证明，当前广受追捧的“被动 Agent 搜索”、“知识编辑 RECIPE”和“供给侧增量 HippoRAG”，在面临真实 Query Drift 时，都在**效率 (Cost)** 或 **效果 (Recall)** 上存在致命缺陷。
2.  **揭示新现象：** 通过硬核诊断数据剖析，首次指出动态 RAG 库更新的核心痛点并非“找不到文档”，而是**“Dense 向量对局部更新知识的多跳语义断层”**。
3.  **提出终极解：** 融合了操作系统缓存淘汰思想（Query-driven Eviction）与图谱连带机制（Graph-assisted），打造出第一个**能在预算约束下，自主预测并适应查询漂移，且解决多跳鸿沟的系统级 RAG 演进框架。**
