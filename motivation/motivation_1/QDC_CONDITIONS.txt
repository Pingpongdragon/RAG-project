QueryDrivenCluster (QDC) — Conditions for Effectiveness
=========================================================

QDC shows a clear advantage over offline maintenance baselines when all
three conditions below hold.

Condition A — Demand signal exists
----------------------------------
q/SF >= 1.3. Multiple queries must reuse the same supporting-fact (SF)
documents, otherwise QDC cannot accumulate enough demand[d] to justify a
replacement.

Condition B — Query text aligns with SF documents
-------------------------------------------------
Query-to-SF semantic alignment >= 0.6. QDC probes the pool by embedding
queries, so query text must directly name or strongly imply the SF entity.
If a query names a movie but the SF is a director page, QDC probes the
wrong documents.

Condition C — Pool contains many distractor documents
-----------------------------------------------------
The pool must contain substantial non-gold distractor documents. QDC's
value is selecting relevant documents amid noise. If nearly every pool
document is useful for some query, then blind baselines such as RandomFIFO
also pick useful documents by chance, shrinking QDC's gap over BestOther.

Mechanism recap
---------------
Each window, QDC:
1. Embeds window queries and probes pool docs, accumulating demand[d].
2. Tracks serve[d] for KB docs that successfully answer queries.
3. Replaces KB doc e with candidate c iff demand[c] > serve[e] + demand[e].

RandomFIFO fairness fix
-----------------------
RandomFIFO now replaces a KB-proportional batch: 0.4% of |KB| per window.
For HotpotQA (KB=10000), this is still 40 docs/window. For FEVER
(KB=3100), this becomes 12 docs/window, preventing H1 self-harm while
preserving the same turnover rate.
